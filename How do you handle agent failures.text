Agent failures are inevitable in production—what differentiates senior systems is not “avoiding failures”, but **designing controlled recovery paths**.

A good mental model:

> An agent system is a distributed system where LLMs, tools, and APIs are unreliable nodes.

So you treat failures like you would in microservices: **detect, classify, retry, degrade, recover, and learn**.

---

# 1. Types of agent failures

## 1.1 Tool failures

* API timeout
* invalid response
* schema mismatch
* rate limit

---

## 1.2 LLM failures

* hallucinated tool calls
* wrong reasoning step
* bad plan generation
* incoherent outputs

---

## 1.3 System failures

* vector DB timeout
* network latency spikes
* context overflow
* memory retrieval failure

---

## 1.4 Execution failures

* infinite loops
* incomplete plans
* partial tool execution
* inconsistent state

---

# 2. Core failure-handling strategy

> Production systems use a **4-layer recovery model**

---

## Layer 1: Detection (fail fast)

You must detect failure immediately:

* tool error codes
* schema validation failure
* timeout thresholds
* repetition detection
* confidence scoring

---

## Layer 2: Classification (what type of failure?)

Example categories:

| Type              | Action            |
| ----------------- | ----------------- |
| transient         | retry             |
| deterministic bug | fix prompt/schema |
| tool outage       | fallback tool     |
| reasoning failure | re-plan           |
| unknown           | escalate          |

---

## Layer 3: Recovery (what do we do?)

---

# 3. Retry strategies (carefully controlled)

## 3.1 Exponential backoff (tool/API failures)

t_{retry} = t_0 \cdot 2^n

Used for:

* rate limits
* network issues
* transient API errors

---

## 3.2 Bounded retries

Never retry infinitely:

```text id="r1"
max_retries = 2 or 3
```

---

## 3.3 Retry with correction (LLM-specific)

If tool call fails:

* feed error back to LLM
* regenerate corrected call

---

# 4. Fallback strategies (VERY important)

When primary path fails:

---

## 4.1 Model fallback

* GPT-4 → GPT-3.5
* large model → small model

---

## 4.2 Tool fallback

* vector DB A → vector DB B
* search API → cached index

---

## 4.3 Cached response fallback

If system fails:

* return last known good answer

---

## 4.4 Graceful degradation

Instead of failing:

* return partial answer
* skip advanced reasoning
* remove tool dependency

---

# 5. Planning-level recovery (agent-specific)

---

## 5.1 Re-planning

If execution fails:

```text id="p1"
Original plan → failure → new plan
```

But controlled:

* max re-plans = 1–2

---

## 5.2 Plan pruning

Remove failed steps and continue.

---

## 5.3 Plan rollback

Return to last valid checkpoint.

---

# 6. State management (critical for agents)

Agents must maintain:

* current step
* tool outputs
* partial results
* execution history

---

## Why?

Without state:

> you cannot recover from partial failures

---

# 7. Circuit breakers (production safety layer)

If failure rate is high:

* stop calling tool
* switch fallback mode
* degrade system

---

## Example:

```text id="c1"
if error_rate > 20%:
    disable_tool()
```

---

# 8. Memory-based recovery

Agents can use memory to recover:

* previous successful strategies
* known working tool patterns
* past failure fixes

---

# 9. Loop failure handling (very common issue)

If agent is stuck:

* detect repetition
* enforce termination
* reset context
* force re-plan

---

## Example detection:

\text{state}*t = \text{state}*{t-k} \Rightarrow \text{loop detected}

---

# 10. Multi-agent failure handling

In multi-agent systems:

* one agent fails → others compensate
* critic agent detects errors
* aggregator filters bad outputs

---

# 11. Observability (absolutely required)

You must log:

* tool call success/failure rate
* LLM retries
* latency per step
* failure category
* recovery path taken

Without this:

> debugging agents becomes impossible

---

# 12. Production architecture for failure handling

```text id="a1"
User Query
   ↓
Agent Execution
   ↓
Failure Detector
   ↓
Failure Classifier
   ↓
Recovery Engine
   ├── Retry
   ├── Re-plan
   ├── Fallback model
   ├── Fallback tool
   ├── Cached answer
   ↓
Final Response / Degraded Response
```

---

# 13. Common mistakes

❌ infinite retries
❌ no fallback system
❌ no failure classification
❌ retrying same prompt unchanged
❌ no circuit breaker
❌ ignoring partial results

---

# 14. Senior-level interview answer

Agent failures in production systems are handled using a layered approach consisting of detection, classification, and controlled recovery. Failures are first detected through tool error codes, timeout thresholds, or validation checks, and then classified as transient, deterministic, or systemic.

Recovery strategies include bounded retries with exponential backoff, model or tool fallback mechanisms, and LLM-based re-planning for execution failures. In addition, circuit breakers and circuit-level monitoring are used to prevent cascading failures, while observability systems track failure rates and recovery effectiveness. This ensures robustness and graceful degradation rather than system collapse.
::`

# 15. Final intuition

Think of agent failure handling like:

| Failure type        | Fix strategy         |
| ------------------- | -------------------- |
| transient API error | retry                |

| infinite loop       | circuit breaker      |
| partial success     | graceful degradation |
