Agents communicate in LLM systems through **controlled data exchange mechanisms**, not free-form “chatting” (in production systems at least). The goal is to make communication **structured, traceable, and cost-bounded**.

A senior framing:

> Agent communication = passing *state, messages, or events* through a defined protocol (not natural language chatter).

---

# 1. Core communication modes

There are 5 primary ways agents communicate:

---

## 1. Message passing (explicit communication)

Agents send structured messages to each other.

```json id="m1"
{
  "from": "planner",
  "to": "retriever",
  "task": "fetch documents on RAG optimization",
  "context": {
    "query": "How to improve retrieval quality?"
  }
}
```

### Characteristics:

* direct agent-to-agent communication
* structured contracts
* predictable flow

✔ used in AutoGen-style systems
✔ easy to trace
❌ can become chatty and expensive

---

# 2. Shared state (blackboard communication)

Agents don’t talk directly—they read/write a shared workspace.

```json id="s1"
{
  "plan": "...",
  "retrieved_docs": "...",
  "partial_solution": "..."
}
```

### Flow:

* Agent A writes
* Agent B reads
* Agent C updates

✔ loosely coupled
✔ scalable
❌ harder to debug conflicts

---

# 3. Event-driven communication

Agents react to system events.

```text id="e1"
Event: "retrieval_complete"
   ↓
Trigger: summarizer agent runs
```

### Event examples:

* tool_completed
* memory_updated
* task_failed
* approval_received (HITL)

✔ highly scalable
✔ asynchronous
❌ complex orchestration

---

# 4. Orchestrator-mediated communication

Most production systems use this.

> Agents do NOT talk directly — the orchestrator routes everything.

```text id="o1"
Planner → Orchestrator → Retriever
Retriever → Orchestrator → Writer
```

### Properties:

* centralized control
* deterministic flow
* strong cost control

✔ production standard
✔ safe
✔ debuggable

---

# 5. Tool-mediated communication (indirect)

Agents communicate via tools + shared outputs.

Example:

* Agent A writes to vector DB
* Agent B retrieves from vector DB

No direct messaging.

---

# 2. Communication topologies

---

## 2.1 Star topology (orchestrator-centered)

```text id="t1"
        Orchestrator
        /    |     \
     A       B      C
```

✔ most common in production
✔ easy control

---

## 2.2 Mesh topology (peer-to-peer)

```text id="t2"
A ↔ B ↔ C ↔ D
```

✔ flexible
❌ hard to control
❌ cost explosion risk

---

## 2.3 Blackboard topology

All agents write to shared memory.

✔ emergent behavior
❌ messy coordination

---

## 2.4 Event bus topology

```text id="t3"
Agents → Event Bus → Agents
```

✔ scalable
✔ decoupled

---

# 3. What is actually being communicated?

Agents don’t exchange “thoughts”—they exchange:

---

## 3.1 State

* partial solutions
* plans
* intermediate outputs

---

## 3.2 Observations

* tool results
* retrieval outputs
* environment feedback

---

## 3.3 Actions

* tool calls
* API requests
* execution commands

---

## 3.4 Metadata

* confidence scores
* token usage
* timestamps
* cost estimates

---

# 4. Communication protocol design (critical in production)

Good systems define:

## 4.1 Schema

```json id="p1"
{
  "task_id": "123",
  "agent": "retriever",
  "input": "...",
  "output": "...",
  "status": "success",
  "confidence": 0.87
}
```

---

## 4.2 Contracts

Each agent expects:

* input format
* output format
* constraints

---

## 4.3 Validation layer

Ensures:

* no malformed outputs
* no hallucinated tool calls
* schema compliance

---

# 5. Communication patterns in real agent systems

---

## 5.1 Sequential communication

```text id="c1"
Planner → Retriever → Writer
```

Simple pipeline.

---

## 5.2 Parallel communication

```text id="c2"
Retriever A ─┐
Retriever B ─┼→ Aggregator
Retriever C ─┘
```

---

## 5.3 Iterative feedback communication

```text id="c3"
Coder → Tester → Debugger → Coder
```

---

## 5.4 Debate / consensus communication

* multiple agents propose answers
* critic evaluates
* aggregator selects best

# 7. Key design constraints

---

## 7.1 Avoid uncontrolled chatter

If agents talk freely:

* cost explodes
* loops form
* unpredictability increases

---

## 7.2 Ensure bounded communication

\text{messages per task} \leq N

---

## 7.3 Ensure determinism where needed

* same input → same communication flow

---

## 7.4 Maintain traceability

Every message should be logged.

---

# 8. Common failures in agent communication

❌ infinite back-and-forth loops
❌ redundant messages between agents
❌ conflicting outputs
❌ missing state synchronization
❌ tool call duplication
❌ inconsistent schemas

---

# 9. Production architecture

```text id="a1"

Orchestrator
   ↓

   ├── Message Passing (agent-agent)
   ├── Shared State (blackboard)

   ↓
Agent Pool
   ↓
Aggregator
   ↓
Final Output
```

# 10. Senior-level interview answer

Agents communicate in multi-agent systems through structured mechanisms such as message passing, shared state (blackboard architectures), event-driven triggers, and orchestrator-mediated routing. Unlike free-form dialogue, production systems enforce strict communication protocols to ensure traceability, determinism, and cost control.

Each communication unit typically contains structured data including task context, intermediate outputs, tool results, and metadata. Depending on the architecture, agents may interact directly in peer-to-peer topologies, indirectly through shared memory, or via a central orchestrator that coordinates execution flow. This ensures scalable and controlled collaboration across multiple specialized agents.
