# Planner–Executor & Reflection Patterns — Senior AI Engineer View

These are two core **agentic AI design patterns** used to make LLM systems more reliable than simple ReAct loops.

They solve a key problem:

> ReAct is flexible, but often unstable. We need structure + verification.

---

# 1. Planner–Executor Pattern

## 1.1 Core idea

Separate reasoning into two roles:

* **Planner** → decides *what to do*
* **Executor** → decides *how to do it*

This decoupling improves control and scalability.

---

## 1.2 Architecture

```text id="p1"
User Query
   ↓
Planner LLM
   ↓
Structured Plan (steps)
   ↓
Executor LLM
   ↓
Tool calls / actions
   ↓
Results
```

---

## 1.3 Example

### Task:

“Write a report on Tesla stock performance and risks”

---

### Step 1: Planner output

```text id="p2"
1. Retrieve Tesla financial data
2. Retrieve recent news about Tesla
3. Analyze stock trends
4. Summarize risks
5. Generate final report
```

Planner does NOT execute anything.

---

### Step 2: Executor

Executes step-by-step:

* calls finance API
* runs web search
* retrieves data
* summarizes

---

## 1.4 Why it works

### Key benefit:

> Planning is separated from execution noise

So:

* fewer hallucinated tool calls
* better long-horizon coherence
* easier debugging

---

## 1.5 Strengths

✔ better structure than ReAct
✔ easier to debug
✔ reusable plans
✔ better for long workflows

---

## 1.6 Weaknesses

❌ brittle if plan is wrong
❌ cannot adapt mid-execution easily
❌ requires re-planning for dynamic environments

---

## 1.7 When to use

* multi-step workflows
* research tasks
* report generation
* code generation pipelines

---

# 2. Reflection Pattern

## 2.1 Core idea

After generating an output, the system:

> **critically evaluates and improves its own output**

This introduces a feedback loop.

---

## 2.2 Architecture

```text id="r1"
Initial Answer
   ↓
Reflection Model (Critic)
   ↓
Error analysis / feedback
   ↓
Improved Answer
```

---

## 2.3 Example

### Step 1: Draft answer

LLM generates response

---

### Step 2: Reflection

Critic model checks:

* factual correctness
* missing information
* logical gaps

---

### Step 3: Revision

LLM rewrites improved answer

---

## 2.4 Reflection loop

A_0 \rightarrow Critique \rightarrow A_1 \rightarrow Critique \rightarrow A_2

Each iteration improves quality.

---

## 2.5 Types of reflection

### (1) Self-reflection

Same model critiques itself

### (2) External reflection

Separate critic model

### (3) Tool-based reflection

* retrieval check
* fact verification
* unit tests (for code)

---

## 2.6 Strengths

✔ reduces hallucination
✔ improves reasoning quality
✔ catches logical errors
✔ improves long-form answers

---

## 2.7 Weaknesses

❌ expensive (multiple passes)
❌ latency heavy
❌ may over-correct (loss of creativity)

---

## 2.8 When to use

* high-stakes outputs (legal, finance, medical)
* code generation
* research summarization
* long-form reasoning tasks

---

# 3. Planner vs Reflection vs ReAct

| Pattern          | Strength             | Weakness           |
| ---------------- | -------------------- | ------------------ |
| ReAct            | flexible tool use    | unstable reasoning |
| Planner–Executor | structured execution | weak adaptation    |
| Reflection       | improves correctness | expensive          |

---

# 4. Combined modern systems (very important)

In production, these are often combined:

## Example architecture:

```text id="m1"
User Query
   ↓
Planner (creates plan)
   ↓
Executor (runs steps via tools)
   ↓
Retriever (RAG context)
   ↓
Generator (draft answer)
   ↓
Reflection (critic check)
   ↓
Final Answer
```

This is basically:

> “agent + RAG + verifier loop”

---

# 5. Key design insight (senior-level)

## Planner = strategy layer

* decides decomposition

## Executor = operational layer

* handles tool interactions

## Reflection = quality control layer

* enforces correctness

---

# 6. Common pitfalls

## ❌ 1. Planner too detailed

→ becomes execution itself (bad separation)

## ❌ 2. Executor improvises plan

→ breaks structure

## ❌ 3. Reflection too strict

→ over-corrects and loses useful info

## ❌ 4. No stopping condition

→ infinite loops or high latency

---

# 7. Production optimizations

## (1) Constrain planner output

Use structured JSON:

```json id="o1"
{
  "steps": [
    "retrieve data",
    "analyze",
    "summarize"
  ]
}
```

---

## (2) Limit reflection rounds

* 1–2 passes usually enough

---

## (3) Selective reflection

Only trigger reflection for:

* high-risk tasks
* low-confidence outputs

---

## (4) Cached plans

Reuse planner output for similar queries

---

# 8. Senior-level interview answer

The Planner–Executor pattern separates high-level task decomposition from execution by using a planning model to generate structured steps and an executor model to carry out tool calls and actions. This improves modularity, debuggability, and long-horizon task handling in agentic systems.

The Reflection pattern introduces a self-critique loop where the model or a separate critic evaluates and improves generated outputs iteratively, reducing factual errors and improving reasoning quality.

In production systems, these patterns are often combined with retrieval and tool use to build robust agentic pipelines that balance structure, adaptability, and output quality.
::`
